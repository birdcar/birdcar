# Implementation Spec: Lead Intake Pipeline — Phase 1

**Contract**: ./contract.md
**Estimated Effort**: M

## Technical Approach

Phase 1 makes the contact form **lossless**. Today the action posts to a `CONTACT_WEBHOOK_URL` (n8n) which doesn't exist yet, so a deploy would silently drop leads. After this phase, every form submit produces a row in D1 — and that row is the source of truth from then on.

Three concrete moves:

1. **Add Drizzle ORM** + drizzle-kit. Define the schema in TS once (`src/db/schema.ts`); drizzle-kit generates the SQL migration into `migrations/`; `wrangler d1 migrations apply` runs it. Schema is the single source of truth — no parallel hand-mirrored TS interface to drift against the SQL.
2. **Provision a D1 database** (`birdcar-leads`) and bind it to the Worker. Apply the auto-generated migration. Stand up a tiny Drizzle client helper that opens a typed connection from a `D1Database` binding.
3. **Rewrite the `contact.send` action** so on validation pass it does a typed `db.insert(leads).values(...)`. Drop the `CONTACT_WEBHOOK_URL`/`CF_ACCESS_*` codepaths entirely. Add two SSR endpoints (`/api/leads/pending`, `/api/leads/[id]`) that n8n polls and patches, both gated by a single bearer token.

n8n's actual workflow lives outside this spec — the gist's `N8N_AI_FLOW.md` describes it. Codebase only needs to expose the contract and trust n8n to honor it.

## Feedback Strategy

**Inner-loop command**: `bun run check`

**Playground**: Astro dev server (`bun run dev`) plus `bun run db:studio` — drizzle-kit's local Studio web UI for inspecting D1 rows during dev.

**Why this approach**: The action and endpoints both run in the worker; the fastest validation is type-check (`astro check` catches Cloudflare runtime + Drizzle type mismatches), then exercise the form via `bun run dev`. drizzle-kit Studio is a much better lead inspector than `wrangler d1 execute --command "SELECT ..."` while iterating.

## Implementation Details

### 1. Add Drizzle dependencies

```sh
bun add drizzle-orm
bun add -D drizzle-kit
```

`drizzle-kit` is dev-only (build-time tool that writes migrations). `drizzle-orm` is runtime — it's tiny and tree-shakes well in Workers bundles.

### 2. drizzle-kit config

**File**: `drizzle.config.ts`

```typescript
import type { Config } from 'drizzle-kit';

export default {
  schema: './src/db/schema.ts',
  out: './migrations',
  dialect: 'sqlite',
  // No `dbCredentials` block needed for migration *generation*;
  // applying happens via wrangler, which reads the binding from wrangler.jsonc.
} satisfies Config;
```

**Key decisions**:
- Output directory is `./migrations/` (not Drizzle's default `./drizzle/`) so `wrangler d1 migrations apply` can find them without extra flags. One canonical location.
- `dialect: 'sqlite'` because D1 speaks SQLite. Drizzle's `d1` driver is for the *runtime* client; `drizzle-kit` itself just writes SQL.

### 3. Schema definition

**File**: `src/db/schema.ts`

```typescript
import { sql } from 'drizzle-orm';
import { sqliteTable, text, integer, index } from 'drizzle-orm/sqlite-core';

export const leads = sqliteTable(
  'leads',
  {
    id: text('id').primaryKey(),                          // uuid v4 from crypto.randomUUID()
    submittedAt: text('submitted_at').notNull(),          // ISO timestamp
    name: text('name').notNull(),
    email: text('email').notNull(),
    message: text('message').notNull(),
    userAgent: text('user_agent'),
    source: text('source').notNull().default('birdcar.dev/contact'),
    status: text('status', {
      enum: ['pending', 'processing', 'done', 'discarded'],
    }).notNull().default('pending'),
    category: text('category', {
      enum: [
        'consulting-fit',
        'consulting-offfit',
        'support-question',
        'vendor-pitch',
        'recruiting',
        'spam',
      ],
    }),
    qualification: text('qualification'),                 // JSON blob (industry/geo/size/problem)
    score: integer('score'),                              // 0..10 from rule-based scorer
    draft: text('draft'),                                 // AI-drafted reply body
    outcome: text('outcome', {
      enum: ['approved', 'edited', 'discarded'],
    }),
    respondedAt: text('responded_at'),                    // ISO timestamp when Nick actually replied
    createdAt: text('created_at')
      .notNull()
      .default(sql`(datetime('now'))`),
    updatedAt: text('updated_at')
      .notNull()
      .default(sql`(datetime('now'))`),
  },
  (table) => ({
    statusIdx: index('idx_leads_status').on(table.status),
    submittedIdx: index('idx_leads_submitted').on(table.submittedAt),
  }),
);

export type Lead = typeof leads.$inferSelect;
export type NewLead = typeof leads.$inferInsert;
```

**Key decisions**:
- Status and category are `text` columns with TS-level enum constraints — Drizzle's enum mode gives compile-time safety without DB-side CHECK constraints (D1 supports CHECK but it's friction for migrations; we trust the type system).
- Timestamps are ISO strings (`text`), not D1's native datetime. Astro Actions already serializes `Date` → ISO; staying string-typed avoids a ser/deser layer.
- `Lead` and `NewLead` types fall out of the table definition. **No more hand-mirrored row interfaces.**
- `qualification` stays as a JSON-encoded string column. If we want typed reads later, add a custom column type.

### 6. Cloudflare env helper

**File**: `src/lib/leads.ts`

This file consolidates env access + Drizzle queries.

```typescript
import { eq, inArray, sql, asc } from 'drizzle-orm';
import type { D1Database, KVNamespace } from '@cloudflare/workers-types';
import { getDb } from '../db/client';
import { leads, type Lead } from '../db/schema';

export interface CloudflareEnv {
  LEADS_DB: D1Database;
  WORKER_PULL_TOKEN?: string;
  SESSION?: KVNamespace;
}

export function getEnv(locals: App.Locals): CloudflareEnv | null {
  return (locals as { runtime?: { env?: CloudflareEnv } }).runtime?.env ?? null;
}

export async function insertLead(
  d1: D1Database,
  input: {
    id: string;
    submittedAt: string;
    name: string;
    email: string;
    message: string;
    userAgent: string | null;
    source?: string;
  },
): Promise<void> {
  const db = getDb(d1);
  await db.insert(leads).values({
    id: input.id,
    submittedAt: input.submittedAt,
    name: input.name,
    email: input.email,
    message: input.message,
    userAgent: input.userAgent ?? null,
    source: input.source ?? 'birdcar.dev/contact',
  });
}

export async function claimPendingLeads(
  d1: D1Database,
  limit = 50,
): Promise<Lead[]> {
  const db = getDb(d1);
  // Two-step claim because D1's SQLite RETURNING semantics inside a subquery
  // are stable but Drizzle's API for "UPDATE WHERE id IN (SELECT ... LIMIT ?)
  // RETURNING *" requires the subquery form. The cleaner Drizzle-y path:
  // 1. Select pending ids in a transaction-equivalent
  // 2. Update those ids to processing, returning rows
  // D1 doesn't support multi-statement transactions yet, so we accept the
  // tiny race window — single n8n poller means it doesn't matter.
  const pending = await db
    .select({ id: leads.id })
    .from(leads)
    .where(eq(leads.status, 'pending'))
    .orderBy(asc(leads.submittedAt))
    .limit(limit);
  if (pending.length === 0) return [];
  const ids = pending.map((row) => row.id);
  const claimed = await db
    .update(leads)
    .set({ status: 'processing', updatedAt: sql`(datetime('now'))` })
    .where(inArray(leads.id, ids))
    .returning();
  return claimed;
}

export async function patchLead(
  d1: D1Database,
  id: string,
  patch: Partial<
    Pick<
      Lead,
      'status' | 'category' | 'qualification' | 'score' | 'draft' | 'outcome' | 'respondedAt'
    >
  >,
): Promise<Lead | null> {
  const db = getDb(d1);
  const updated = await db
    .update(leads)
    .set({ ...patch, updatedAt: sql`(datetime('now'))` })
    .where(eq(leads.id, id))
    .returning();
  return updated[0] ?? null;
}
```

**Key decisions**:
- `claimPendingLeads` uses two queries (select + update). D1 doesn't support multi-statement transactions yet, so we can't atomically `UPDATE ... WHERE id IN (SELECT ...)` through Drizzle's builder. The race window is small and only matters with multiple pollers — single n8n instance means single-poller. If we ever scale to 2+ pollers, swap this for a raw `db.run(sql\`...\`)` UPDATE-with-subquery.
- `patchLead` returns `Lead | null` — Drizzle's `.returning()` gives an array; we slice the first row.
- All SQL `now()` calls use `sql\`(datetime('now'))\`` — Drizzle's parameterized SQL helper, not a string concat.

### 7. Rewrite `contact.send` action

**File**: `src/actions/index.ts`

**Pattern to follow**: existing action structure (don't restructure the file; just swap the handler body).

```typescript
import { ActionError, defineAction } from 'astro:actions';
import { z } from 'astro:schema';
import { getEnv, insertLead } from '../lib/leads';

export const server = {
  contact: {
    send: defineAction({
      accept: 'form',
      input: z.object({
        name: z.string().trim().min(1, 'Tell me your name.').max(120),
        email: z.string().trim().email('Give me an email I can reply to.').max(254),
        message: z.string().trim().min(10, 'A sentence or two helps me reply usefully.').max(5000),
      }),
      handler: async (input, ctx) => {
        const env = getEnv(ctx.locals);
        if (!env?.LEADS_DB) {
          throw new ActionError({
            code: 'INTERNAL_SERVER_ERROR',
            message: 'The contact form is not configured. Email me directly at hi@birdcar.dev.',
          });
        }
        const id = crypto.randomUUID();
        const submittedAt = new Date().toISOString();
        try {
          await insertLead(env.LEADS_DB, {
            id,
            submittedAt,
            name: input.name,
            email: input.email,
            message: input.message,
            userAgent: ctx.request.headers.get('user-agent'),
            source: 'birdcar.dev/contact',
          });
        } catch {
          throw new ActionError({
            code: 'INTERNAL_SERVER_ERROR',
            message: "I couldn't save that. Email hi@birdcar.dev directly and I'll see it.",
          });
        }
        return { delivered: true, id } as const;
      },
    }),
  },
};
```

### 8. Pending leads endpoint

**File**: `src/pages/api/leads/pending.ts`

```typescript
import type { APIRoute } from 'astro';
import { claimPendingLeads, getEnv } from '../../../lib/leads';

export const prerender = false;

function unauthorized(): Response {
  return new Response(JSON.stringify({ error: 'unauthorized' }), {
    status: 401,
    headers: { 'content-type': 'application/json' },
  });
}

function isAuthorized(request: Request, expected: string | undefined): boolean {
  if (!expected) return false;
  const header = request.headers.get('authorization');
  if (!header?.startsWith('Bearer ')) return false;
  return header.slice('Bearer '.length).trim() === expected;
}

export const GET: APIRoute = async ({ request, locals }) => {
  const env = getEnv(locals);
  if (!env || !isAuthorized(request, env.WORKER_PULL_TOKEN)) {
    return unauthorized();
  }
  const url = new URL(request.url);
  const limitRaw = url.searchParams.get('limit');
  const limit = Math.min(Math.max(Number.parseInt(limitRaw ?? '50', 10) || 50, 1), 200);
  const leads = await claimPendingLeads(env.LEADS_DB, limit);
  return new Response(JSON.stringify({ leads }), {
    status: 200,
    headers: { 'content-type': 'application/json', 'cache-control': 'no-store' },
  });
};
```

### 9. Lead patch endpoint

**File**: `src/pages/api/leads/[id].ts`

```typescript
import type { APIRoute } from 'astro';
import { z } from 'astro:schema';
import { getEnv, patchLead } from '../../../lib/leads';

export const prerender = false;

const PatchSchema = z.object({
  status: z.enum(['pending', 'processing', 'done', 'discarded']).optional(),
  category: z.enum([
    'consulting-fit',
    'consulting-offfit',
    'support-question',
    'vendor-pitch',
    'recruiting',
    'spam',
  ]).nullable().optional(),
  qualification: z.string().max(8000).nullable().optional(),
  score: z.number().int().min(0).max(10).nullable().optional(),
  draft: z.string().max(10000).nullable().optional(),
  outcome: z.enum(['approved', 'edited', 'discarded']).nullable().optional(),
  respondedAt: z.string().datetime().nullable().optional(),
});

function jsonResponse(status: number, body: unknown): Response {
  return new Response(JSON.stringify(body), {
    status,
    headers: { 'content-type': 'application/json', 'cache-control': 'no-store' },
  });
}

function isAuthorized(request: Request, expected: string | undefined): boolean {
  if (!expected) return false;
  const header = request.headers.get('authorization');
  return Boolean(header?.startsWith('Bearer ') && header.slice(7).trim() === expected);
}

export const PATCH: APIRoute = async ({ request, params, locals }) => {
  const env = getEnv(locals);
  if (!env || !isAuthorized(request, env.WORKER_PULL_TOKEN)) {
    return jsonResponse(401, { error: 'unauthorized' });
  }
  const id = params.id;
  if (!id || typeof id !== 'string') {
    return jsonResponse(400, { error: 'missing id' });
  }
  let body: unknown;
  try {
    body = await request.json();
  } catch {
    return jsonResponse(400, { error: 'invalid json' });
  }
  const parsed = PatchSchema.safeParse(body);
  if (!parsed.success) {
    return jsonResponse(400, { error: 'invalid body', issues: parsed.error.issues });
  }
  const updated = await patchLead(env.LEADS_DB, id, parsed.data);
  if (!updated) {
    return jsonResponse(404, { error: 'lead not found' });
  }
  return jsonResponse(200, { lead: updated });
};
```

## Data Model

## API Design

| Method | Path | Auth | Purpose |
|---|---|---|---|
| `POST` | `/_actions/contact.send` | none (Astro Action handles its own gates) | Form submit → INSERT |
| `GET` | `/api/leads/pending?limit=N` | `Bearer $WORKER_PULL_TOKEN` | Claim + return up to N pending leads |
| `PATCH` | `/api/leads/:id` | `Bearer $WORKER_PULL_TOKEN` | Update lead fields |

## Failure Modes

| Component | Failure | Behavior |
|---|---|---|
| `contact.send` | D1 unreachable | Action throws `INTERNAL_SERVER_ERROR` with the existing fail-soft "Email hi@birdcar.dev" message |
| `contact.send` | Validation fails | Existing field-level error rendering (already shipped) |
| `claimPendingLeads` | Pending select returns 0 | Endpoint returns `{ leads: [] }` |
| `claimPendingLeads` | Update fails after select | Stale `processing` row that never advances. Practical fix: a Cron-triggered "reset stuck processing rows older than 5min back to pending" sweep — defer to Phase 2 unless it bites |
| `GET /api/leads/pending` | No bearer token | 401 |
| `PATCH /api/leads/:id` | Bad body | 400 with Zod issues |
| `PATCH /api/leads/:id` | Lead not found | 404 |
